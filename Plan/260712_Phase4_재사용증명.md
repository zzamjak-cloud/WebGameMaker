# WebGameMaker Phase 4 재사용 증명 계획

## 1. 목표

Phase 4의 목표는 첫 게임 `수문 07 — 마지막 등불`에서 추출한 공용 모듈과 스키마가 두 번째 탑다운 게임 제작에 실제로 재사용되는지 증명하는 것이다.

완료 게이트:

- 두 번째 탑다운 게임을 `games/relay-ward`에 추가한다.
- 두 게임이 `module.player-move-2d`, `module.camera-follow`, `module.health`, `module.enemy-patrol`, `module.enemy-chase`, `module.collision-layer`를 JSON module binding으로 참조한다.
- 두 번째 게임은 첫 게임의 `src/features`를 import하지 않는다.
- 게임 간 20줄 이상 동일한 게임별 로직 블록이 없어야 한다.
- Phase 1 기준선 대비 두 번째 게임의 신규 게임별 gameplay LOC와 수동 설정 단계가 각각 50% 이상 감소해야 한다.
- 루트 `pnpm verify`가 두 게임 검증과 Phase 4 재사용 비교를 포함해 통과해야 한다.

## 2. 두 번째 게임 방향

제목은 `Relay Ward — 마지막 전송`으로 둔다. 같은 탑다운 액션 조작과 적 순찰/추적을 쓰되, 목표는 적 제거가 아니라 세 개의 relay node를 순서대로 동기화해 중앙 송신기를 깨우는 것이다.

차이점:

- 첫 게임은 조사광으로 적을 제압하고 신호등을 점등한다.
- 두 번째 게임은 짧은 pulse로 relay node를 동기화하며, 적 접촉과 node 순서 오류가 위험 요소다.
- 적은 동일한 patrol/chase/contact 공용 모듈 설정을 사용하지만 위치·속도·감지 거리가 다르다.

## 3. 코드 경계

- 공용 이동·체력·적 AI·접촉 피해 계산은 `@web-game-maker/core-modules`에서 직접 사용한다.
- 두 번째 게임에는 프로젝트 컴파일, 상태 전이, relay 규칙처럼 게임 고유 규칙만 둔다.
- 첫 게임 패키지에서 소스 import를 금지한다.
- 런타임 UI 전환은 Phase 4의 핵심이 아니므로 기본 앱은 기존 수직 절편을 유지하고, 두 번째 게임은 패키지·schema·측정·검증 대상으로 추가한다.

## 4. 측정

Phase 1 기준선은 `tests/benchmarks/reuse-baseline.phase-1.json`의 `commonGameplayLoc=390`, `manualSetupSteps=10`을 원본으로 둔다.

Phase 4 측정은 `games/relay-ward/src/gameplay`와 `games/relay-ward/reuse-recipe.json`을 대상으로 한다.

통과 기준:

- `commonGameplayLoc <= 195`
- `manualSetupSteps <= 5`
- 측정 결과는 `tests/benchmarks/reuse-baseline.phase-4.json`에 고정한다.

## 5. 검증

- `pnpm --filter @web-game-maker/relay-ward test`
- `pnpm wgm validate games/relay-ward`
- `pnpm measure:reuse -- --source-root games/relay-ward/src/gameplay --recipe games/relay-ward/reuse-recipe.json --baseline tests/benchmarks/reuse-baseline.phase-4.json --check`
- `pnpm test:benchmarks`
- `pnpm verify`
